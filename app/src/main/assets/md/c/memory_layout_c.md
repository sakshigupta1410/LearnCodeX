# **Memory Layout of a C Program**

---

When a C program is executed, its process memory is divided into several segments, each with a specific role in the program’s runtime behavior. Understanding these segments is key to mastering memory management, debugging, and writing efficient C code.

---

## **1. Text (Code) Segment**

- **What it is:**  
  This segment contains the **machine code instructions** generated by the compiler — essentially the program itself.

- **Characteristics:**
    - Typically **read-only** to prevent accidental modification, improving security.
    - Loaded into memory when the program starts.
    - Usually shared among multiple running instances of the program to save memory.

- **Why it matters:**  
  Any attempt to modify this segment (e.g., trying to write to a function’s code) leads to a **segmentation fault**.

---

## **2. Data Segment**

The data segment is divided into two key parts:

### **Initialized Data Segment**

- Stores **global** and **static variables** that are explicitly initialized in the code.
- These variables exist for the **entire lifetime** of the program.
- Example:

```c
  int globalVar = 10;      // Stored in initialized data segment
  static int staticVar = 20;
```

### **Uninitialized Data Segment (BSS)**

Contains **global** and **static** variables that are not initialized or initialized to zero.

**BSS** stands for “**_Block Started by Symbol_**”.

Variables here also exist for the program’s lifetime and are initialized to zero by default.

**Example**:

```
int uninitializedGlobal;  // Stored in BSS segment
static int staticZeroInit; // Also in BSS

```

---

## **3. Heap**

* The **heap** is a region of memory used for dynamic memory allocation during program execution.

* It grows upwards (**toward higher memory addresses**) as the program requests more space (using functions like malloc(), calloc(), realloc()).

* The programmer is responsible for managing heap memory: allocating and freeing it to avoid memory leaks.

**Example**:

```
int *ptr = malloc(sizeof(int) * 5);  // allocate memory for 5 integers on heap
if (ptr != NULL) {
// use the memory
free(ptr);  // free the allocated memory when done
}
```

**Important**: Failure to free heap memory leads to leaks and inefficient use of resources.

## **4. Stack**

* The stack stores **local variables**, **function parameters**, and return addresses.

* It grows **downwards** (towards lower memory addresses).

* Each time a function is called, a **stack frame** is created to hold its local variables and control information.

* When the function **returns**, its stack frame is popped and memory reclaimed automatically.

**Characteristics**:

- Very fast memory **allocation** and **deallocation**.

- Size is usually **limited**; large allocations should avoid the stack.

**Example**:

```
void func() {
int localVar = 5;  // stored on stack
// ...
}
```

## **Memory Layout Diagram (Conceptual)**

| Area              | Purpose                                 | Managed By        |
|-------------------|------------------------------------------|-------------------|
| Stack             | Function calls, local vars               | JVM               |
| Heap              | Objects, classes, arrays                 | Garbage Collector |
| Code              | Compiled bytecode                        | Android Runtime   |
| Native (C/C++)    | Native libs, JNI, NDK                    | OS (Linux kernel) |
| Graphics Memory   | Bitmaps, surfaces                        | GPU / Skia        |

---

### **Why Understanding Memory Layout is Important**

**Debugging**: Knowing where variables live helps fix issues like segmentation faults, stack overflows, or memory corruption.

**Optimization**: Efficient memory usage and avoiding unnecessary allocations improve performance.

**Security**: Prevent buffer overflows and exploit mitigations by understanding stack vs heap.

**Multithreading**: Some segments like stack are per-thread, heap and globals are shared, affecting concurrency design.

---

## **Common Issues Related to Memory Segments**
<br>

### **Stack Overflow**

* Happens if a program uses too much stack space (e.g., deep or infinite recursion, large local arrays).

* Usually causes a crash or segmentation fault.

### **Memory Leak**

* Occurs when dynamically allocated memory on the heap is not freed properly.

* Can cause the program to consume increasing amounts of memory over time.

## 🧪 **Practice Challenge**

1. Write a program that:

2. Declares global, static, and local variables.

3. Allocates memory dynamically on the heap.

4. Prints the memory addresses of all these variables.

5. Observe which variables are close in memory and their segment locations.

**Example starter:**

```
#include <stdio.h>
#include <stdlib.h>

int global_var = 5;              // initialized global
static int static_var;           // uninitialized static
int main() {
int local_var = 10;          // local variable (stack)
int *heap_var = malloc(sizeof(int));  // dynamically allocated variable (heap)

    printf("Global variable address: %p\n", &global_var);
    printf("Static variable address: %p\n", &static_var);
    printf("Local variable address: %p\n", &local_var);
    printf("Heap variable address: %p\n", heap_var);
    
    free(heap_var);
    return 0;
}
```

Run it multiple times and compare addresses!
